## Decisões registradas (ADR resumido)

- **Monólito modular, não microserviços** — evita complexidade operacional prematura; módulos NestJS já mapeiam para os domínios do documento mestre (Identity, Contacts, Channels, ...), o que permite extração futura se necessário.
- **RBAC como allow-list explícita por rota, sem hierarquia implícita de papéis** — Agent e Analyst têm permissões de natureza diferente de Builder, não um subconjunto; listar papéis explicitamente por rota evita bugs de "Owner esqueceu de herdar uma permissão nova".
- **Um único app Meta para todos os workspaces** — a assinatura do webhook (`X-Hub-Signature-256`) usa o App Secret da plataforma, não por conexão; o roteamento por workspace acontece depois, via `externalAccountId` (ID da página/conta IG), que é global e único por provider.
- **Conexão de canal manual neste marco** — o fluxo OAuth completo com a Meta (App Review, permissões, troca de código por token) é um projeto à parte; por ora a conexão aceita um token já obtido, para poder testar o adapter e o webhook de ponta a ponta sem depender de aprovação externa.
- **Outbox publicado por poller em memória (`setInterval`), não BullMQ** — Redis/BullMQ só entram no M6 junto da fila de execução do runtime; introduzir essa infra só para o outbox seria uma dependência operacional sem benefício neste marco. O poller processa em lotes de 20 a cada 2s, marca `processedAt` só após o `EventEmitter2.emitAsync` ter sucesso, e desiste (loga e marca como processado mesmo assim) depois de 5 tentativas para não travar a fila atrás de um evento "veneno".
- **`contact.message_received` é publicado dentro da mesma transação que cria/atualiza o contato** — antes (M3) a criação do contato e o `InboundEvent` já eram atômicos entre si; agora o `OutboxEvent` também entra nesse mesmo `$transaction` em `ChannelsService.ingestEvent`, então nunca existe um estado em que o contato foi criado mas o evento não foi publicado (ou vice-versa).
- **Grafo da automação como um único `Json` versionado, não `automation_nodes`/`automation_edges` normalizados** — a versão publicada precisa ser barata de carregar inteira para o runtime (M6); normalizar em tabelas adicionaria joins sem benefício até existir uma UI que edite nó a nó (M7). Migrar para tabelas é uma mudança isolada nessa camada depois, não um retrabalho do runtime.
- **Publicar sempre abre um novo draft (cópia) e nunca muta a versão publicada** — `AutomationsService.publish` marca a versão atual como `PUBLISHED` (arquivando a anterior) e imediatamente cria uma nova linha `DRAFT` com o mesmo grafo; edições subsequentes (`updateDraft`) só tocam essa nova linha, então uma versão publicada nunca muda depois que uma execução começou a rodar contra ela.
- **Cada job do BullMQ processa exatamente um nó do grafo, não a automação inteira** — `ExecutionRunnerService.runStep` executa um passo e decide se reenfileira (imediato ou com delay), pausa (`WAITING`) ou termina; isso torna o limite de passos, o registro por passo (`AutomationStepExecution`) e a semântica de retry triviais de raciocinar, ao custo de mais idas ao banco do que processar tudo numa única invocação.
- **Lógica de execução separada do adapter BullMQ** — `ExecutionRunnerService` não conhece BullMQ (testável sem Redis); `ExecutionProcessor` é só um adaptador fino que chama `runStep` e move o job para a dead-letter queue quando as tentativas se esgotam. `docker-compose.yml` já sobe Redis, mas o ambiente de desenvolvimento atual não conseguiu validar isso contra uma fila real (mesma limitação de sandbox do Postgres) — os 98 testes cobrem a lógica com Prisma/BullMQ mockados, não o comportamento real de fila.
- **`MetaAdapter.sendMessage` agora chama a Graph API de verdade** (`POST /{page-id}/messages` com `Authorization: Bearer`), mas nunca foi exercitado contra a API real da Meta — só testado com `fetch` mockado. Precisa de credenciais reais de um app Meta aprovado para validar de ponta a ponta.
- **`human_handoff` apenas pausa a execução (`WAITING`) neste marco** — não existe Conversation/Inbox ainda (chega no M8); a automação fica parada nesse nó até um marco futuro implementar a retomada.
- **M0-M6 validados de ponta a ponta contra infraestrutura real** (2026-07-17): Docker não sobe neste sandbox, mas um Postgres 14 e um Redis locais (via `initdb`/`redis-server` diretos, fora do `docker-compose.yml`) permitiram rodar a API de verdade e confirmar signup → workspace → canal → automação publicada → webhook assinado → outbox → execução → tentativa real contra a Graph API da Meta (retornou 401 esperado, já que o token de teste não é real) → 5 tentativas com backoff → `FAILED_PERMANENT` → job na dead-letter queue. Replay do mesmo webhook confirmou zero duplicação (contato, execução e outbox ficaram em 1 cada). Isso fecha a lacuna de validação que vinha sendo carregada desde o primeiro marco.
- **M7 (Flow Builder) também validado no navegador contra a API real, não só com testes automatizados** (2026-07-17): fluxo completo via browser — signup → criar workspace → criar automação → montar o grafo arrastando nós (gatilho, mensagem, fim) e conectando arestas no canvas React Flow → "Validar" retornou erros reais do `graph-validator` (nós desconectados) → após conectar corretamente, "Validar" confirmou "Grafo válido" → "Publicar" criou a versão 1 no Postgres real. Nenhum mock em nenhuma camada (frontend → proxy Next.js → API NestJS → Postgres).
- **Autenticação do frontend via cookie httpOnly, nunca localStorage** — `POST /api/auth/login` do Next.js troca credenciais pela API e grava o JWT num cookie `mz_token` httpOnly; todo fetch autenticado do lado do cliente passa por `/api/proxy/[...path]`, que lê o cookie no servidor e injeta o header `Authorization` — o token nunca fica acessível a JS no browser.
- **Deploy dividido: API no Render, frontend na Vercel** — a API precisa ficar sempre ligada (o worker BullMQ processa a fila continuamente, inclusive nós de espera agendados), o que não cabe no modelo serverless da Vercel; o frontend Next.js, por ser request-driven, roda bem na Vercel, que tem suporte nativo melhor que o Render pra isso. `render.yaml` cobre só a API; os passos da Vercel estão documentados no README.
- **M8 (Inbox + handoff humano) implementado e validado ponta a ponta contra infraestrutura real** (2026-07-19): novos modelos `Conversation`/`ConversationMessage` (`apps/api/prisma/schema.prisma`), módulo `apps/api/src/conversations/` (CRUD + `claim`/`close`/`resume`, RBAC via `WorkspaceRolesGuard`), e o nó `human_handoff` do runtime agora abre/reaproveita uma conversa de verdade (`ExecutionRunnerService.runHumanHandoff` → `ConversationsService.openForHandoff`) em vez de só pausar em `WAITING`. `resume` reenfileira a `AutomationExecution` que ficou esperando no BullMQ, exatamente como um nó de delay retomando. Testado via HTTP real contra Postgres/Redis/BullMQ locais: webhook assinado → automação com `human_handoff` → conversa criada `WAITING_HUMAN` → agente `claim` → tentativa de resposta manual bateu de verdade em `graph.facebook.com` (401 esperado, token de teste) → `resume` → execução voltou a `QUEUED` e completou (`COMPLETED`) — e no Flow Builder + Inbox reais no navegador (login → assumir → responder → fechar).
- **Mensagens de conversa alimentadas por listener de evento, não pela transação do webhook** — `ConversationsEventListener` escuta `contact.message_received` (o mesmo evento que `TriggerMatcherService` já consome) em vez de `ChannelsService.ingestEvent` gravar `ConversationMessage` na mesma transação do webhook. Evita um ciclo de módulos NestJS (`ChannelsModule` ↔ `ConversationsModule`, já que `ConversationsService` precisa do `MetaAdapter`/`CredentialsCipher` de `ChannelsModule` para respostas manuais) e segue o desenho já estabelecido de outbox/eventos para efeitos colaterais desacoplados da escrita principal.
- **M9 (Policy Engine) implementado e validado ponta a ponta contra infraestrutura real** (2026-07-19): `PolicyService.canSend(contactId, channelConnectionId, provider)` (`apps/api/src/policy/`) é o único ponto de decisão — checa, nessa ordem, opt-out (`Contact.optedOutAt`), consentimento revogado (`ContactConsent` com `purpose="messaging"` para o canal) e a janela de 24h (`MetaAdapter.getMessagingWindow` contra a última `ConversationMessage` `IN` real do contato nesse canal, não um campo derivado que podia ficar dessincronizado). Chamado tanto por `ExecutionRunnerService.runSendMessage` (nó `send_message` do runtime) quanto por `ConversationsService.sendMessage` (resposta manual no Inbox), então a regra existe uma única vez. Uma negação no runtime é permanente (`failPermanently`), nunca retry - esperar não reabre a janela nem desfaz um opt-out.
- **Opt-out/opt-in por palavra-chave, não NLP** — `OptOutListener` (`apps/api/src/policy/opt-out.listener.ts`) reage a `contact.message_received`, casando a mensagem contra uma lista fixa de palavras (`parar`/`pare`/`sair`/`cancelar`/`stop`/`unsubscribe` para opt-out; `voltar`/`iniciar`/`start`/`subscribe` para opt-in) como palavra isolada (regex com fronteira de palavra), não substring nem classificação de intenção. Ao opt-out, cancela toda `AutomationExecution` não-terminal desse contato (`QUEUED`/`RUNNING`/`WAITING`/`FAILED_RETRYABLE` → `CANCELED`) - inclusive uma automação que a própria mensagem de opt-out acabou de disparar, confirmado no teste de ponta a ponta abaixo.
- **Convenção nova, não imposta**: `purpose="messaging"` em `ContactConsent` é uma convenção introduzida pelo Policy Engine, não uma migration - se nenhum registro de consentimento existir para um contato/canal, o envio não é bloqueado por esse motivo (o gate de consentimento é opt-in por design; a Meta já impõe a janela de 24h como controle real). Cabe ao workspace registrar `POST /contacts/:id/consents` com esse `purpose` se quiser que revogações bloqueiem envios.
- **Validado contra infraestrutura real, não só os 122 testes unitários** (2026-07-19): com o servidor rodando contra Postgres/Redis/BullMQ reais, uma automação `trigger → send_message → end` publicada chegou a tentar `graph.facebook.com` de verdade (5 tentativas, 401 esperado) antes do Policy Engine ter qualquer motivo para bloquear; mandar "PARAR" do mesmo contato setou `optedOutAt`, cancelou a execução `WAITING` de handoff em andamento, e a automação dessa mesma mensagem "PARAR" falhou permanentemente em 1 passo (não 5 tentativas) com "Contact has opted out of messaging" - provando que o bloqueio aconteceu antes de qualquer chamada de rede. Mandar "voltar" limpou `optedOutAt`. No Inbox, resposta manual funcionou dentro da janela (chegou a bater na Graph API), foi bloqueada com 403 claro depois de eu atrasar manualmente a última mensagem `IN` para 25h atrás, e voltou a ser bloqueada (403 diferente) depois de revogar um consentimento `purpose="messaging"` via API.
- **Nós `action`, `goal` e `start_another_flow` implementados no runtime** (2026-07-19, `apps/api/src/automations/execution/execution-runner.service.ts`): `action` suporta `add_tag`/`remove_tag`/`set_field` via chamadas diretas ao Prisma (mesmo padrão de `ContactsService`, sem importar o módulo inteiro para evitar acoplamento); `goal` é um nó de passagem puro (marca e avança); `start_another_flow` cria uma nova `AutomationExecution` fire-and-forget para outra automação publicada, deduplicada por `triggerEventId = start_another_flow:${execution.id}:${node.id}` (mesma técnica de idempotência do `TriggerMatcherService`) e com uma guarda contra o auto-loop mais óbvio (uma automação apontando pra si mesma) - um ciclo indireto A→B→A não é detectado, limitação registrada no README.
- **Corrigida uma inconsistência real entre validador e runtime, descoberta ao implementar M8**: `human_handoff` passou a exigir aresta de saída no runtime desde o M8 (`ExecutionRunnerService.runHumanHandoff` falha permanentemente sem uma), mas `graph-validator.ts` ainda tinha uma exceção explícita permitindo esse nó sem aresta de saída, e o Flow Builder (`AutomationNode.tsx`) nem desenhava o handle de saída para ele - um grafo publicável passaria na validação e falharia só em produção. Removida a exceção no validador, adicionado o handle no componente visual, e o teste correspondente em `graph-validator.spec.ts` reescrito para cobrir os dois sentidos (com e sem aresta).
- **Validado contra infraestrutura real** (2026-07-19): publiquei `trigger → action(add_tag "vip") → goal → start_another_flow(outra automação publicada) → end` e mandei um webhook de verdade - o contato ganhou a tag no Postgres, o passo do goal foi registrado, e o `start_another_flow` criou e enfileirou uma nova `AutomationExecution` real que por sua vez chegou a tentar `graph.facebook.com` de verdade no seu próprio `send_message`. No Flow Builder, adicionar o nó pela paleta, editar no painel de propriedades, e "Validar" contra a API real confirmaram as novas regras (`ACTION_MISSING_TYPE`, `ACTION_MISSING_TAG`, `START_ANOTHER_FLOW_MISSING_TARGET`) funcionando de ponta a ponta, não só no tipo do TypeScript do frontend.
- **Nós `collect_input` e `external_request` implementados no runtime, fechando a lista de tipos de nó** (2026-07-20): `collect_input` pausa (`WAITING`) sem pré-avançar `currentNodeId` - diferente de `human_handoff`/`delay`, que já sabem para onde ir ao retomar, aqui a retomada precisa decidir isso *e* gravar o valor coletado, então fica pra `CollectInputListener` (novo, reage a `contact.message_received` como `TriggerMatcherService`/`ConversationsEventListener`/`OptOutListener` já fazem) fazer as duas coisas atomicamente quando a próxima mensagem do contato chega. O valor coletado vai para `AutomationExecution.contextJson` e fica disponível como `{{flow.<variableName>}}` - novo namespace em `variable-resolver.ts`, com `graph-validator.ts` checando que toda referência a `flow.*` corresponda a um `collect_input` de fato presente no grafo (mesmo tratamento que já existia para `contact.custom.*`).
- **`external_request` falha fechado por padrão (SSRF)**: `EXTERNAL_REQUEST_ALLOWED_HOSTS` (env var, hosts separados por vírgula) precisa listar o host explicitamente ou toda chamada falha antes de qualquer request de rede - vazio/não setado bloqueia tudo. Essa é uma allow-list global do operador do deploy, não por workspace; um Builder dentro de um workspace pode digitar qualquer URL num nó, mas ela só sai de fato se o operador já tiver liberado aquele host. Timeout via `AbortController` (10s por padrão, configurável por nó via `timeoutMs`), retry em falha via o mesmo padrão do `send_message` (`FAILED_RETRYABLE` + rethrow pro BullMQ), e resposta salva num campo personalizado via `saveResponseAs` (JSON parseado quando possível, texto puro como fallback) - reaproveitando `setCustomFieldValue`, extraído do `action` `set_field` pra não duplicar a lógica de coerção/upsert.
- **Validado contra infraestrutura real, incluindo uma chamada HTTPS de saída de verdade** (2026-07-20): publiquei `trigger → collect_input(favorite_color) → external_request(POST https://postman-echo.com/post) → send_message → end`. Webhook 1 (gatilho) → execução ficou `WAITING` com `currentNodeId` ainda no nó `collect_input` (confirmando que não pré-avança). Webhook 2 ("roxo") → `CollectInputListener` gravou `{"favorite_color": "roxo"}` em `contextJson`, avançou, e o `external_request` bateu de verdade em `postman-echo.com`, recebeu 200, e salvou a resposta JSON num campo `JSON` (uma tentativa anterior com um campo `TEXT` falhou com erro de coerção claro - confirmando que a chamada de rede realmente aconteceu antes da falha). `send_message` renderizou `{{flow.favorite_color}}` → "You picked roxo" corretamente. Uma segunda automação apontando para um host fora da allow-list (`evil.example.com`) falhou permanentemente sem nenhuma tentativa de rede, confirmando o fail-closed.
- **M10 (Observabilidade) implementado** (2026-07-20): `GET /health` (`apps/api/src/app.controller.ts`) agora checa Postgres (`SELECT 1`) e Redis (`client.info()` via a conexão do BullMQ - a interface abstrata do client do BullMQ não expõe `ping()`, só comandos que ele mesmo usa) de verdade, retornando 503 com o detalhamento de qual dependência caiu, em vez de sempre `{status: "ok"}` só por o processo estar de pé - é o `healthCheckPath` que o Render já apontava desde o `render.yaml` original, só que sem checar nada até agora. `GET /metrics` (`apps/api/src/observability/`) expõe contagem de execuções por status, profundidade da fila de execução e da dead-letter queue (via `Queue.getJobCounts()`), backlog do outbox (contagem + idade do mais antigo não processado) e total de negações do Policy Engine (contagem de `AutomationStepExecution` cujo erro bate com uma das mensagens de `POLICY_DENIAL_MESSAGES`) - snapshot JSON simples, não formato Prometheus, porque não existe scrape target/stack de monitoramento configurado ainda; seria infraestrutura especulativa. Protegido por `METRICS_TOKEN` no header `X-Metrics-Token`, falha fechado como o `EXTERNAL_REQUEST_ALLOWED_HOSTS` (token vazio/não setado nunca autoriza).
- **Logs estruturados em JSON só em produção** (`apps/api/src/common/logging/json-logger.service.ts`): `JsonLogger` implementa `LoggerService` do Nest e escreve uma linha JSON por chamada (parseável por agregadores de log como o stream do Render), ligado via `NODE_ENV=production` em `main.ts` - o dev local continua com o logger colorido padrão do Nest, mais legível no terminal. Como as assinaturas de `log`/`error`/etc. do Nest são variádicas e ambíguas por design (`log(msg, context)` vs `error(msg, stack, context)`), um único parâmetro string é rotulado `context` (o caso mais comum no código já existente); dois ou mais viram um array `details` verbatim, em vez de arriscar rotular errado qual string é o quê.
- **Caminho de falha do `/health` coberto só por teste unitário, não pela mesma validação de infraestrutura real dos outros itens deste marco** (2026-07-20): provocar uma queda de verdade do Postgres/Redis locais para testar o branch 503 arriscaria desestabilizar o ambiente de dev usado pelo resto da sessão; o caminho de sucesso foi validado contra Postgres/Redis/BullMQ reais (`curl /health` → 200, `curl /metrics` com token → dados reais refletindo toda a atividade da sessão), e o caminho de falha (DB ou Redis fora do ar) tem cobertura de teste unitário exercitando exatamente o mesmo código (`app.controller.spec.ts`), não uma reimplementação em mock.
- **OAuth automático de canal implementado, escopo deliberadamente reduzido ao caso comum** (2026-07-22, `apps/api/src/channels/meta-oauth.service.ts` + `apps/web/src/app/api/oauth/meta/`): troca `code` por token de longa duração, busca `/me/accounts`, e conecta automaticamente só quando existe **exatamente uma Page** - reaproveitando 100% do `ChannelsService.connect` já existente (mesma criptografia, mesmo dedupe, mesmo audit log). Zero ou mais de uma Page retorna um erro claro pedindo o fluxo manual, em vez de construir um cache de estado pendente + uma tela de escolha de Page - decisão consciente de simplicidade a pedido do usuário ("simplifique o máximo possível, desde que continue funcionando"), documentada como limitação no README em vez de ser um projeto à parte.
- **CSRF do OAuth via cookie httpOnly + nonce simples, sem assinatura criptográfica**: `start` gera um nonce aleatório e grava `${nonce}.${workspaceId}` num cookie httpOnly `sameSite=lax` (precisa de `lax`, não `strict`, porque o cookie tem que sobreviver a uma navegação de topo vinda de `facebook.com`); `callback` exige que o `state` recebido bata exatamente com o nonce do cookie antes de confiar no `workspaceId` embutido nele. Não há JWT nem HMAC aqui - o cookie já é a fonte de verdade (só o servidor escreve nele), então uma comparação direta é suficiente e mais simples.
- **O callback fica no Next.js, não na API** - só o Next.js tem acesso ao cookie httpOnly da sessão numa navegação `GET` direta do browser; a troca de `code` por token (que usa `META_APP_SECRET`) acontece na API, chamada server-to-server pelo Next.js exatamente como `serverApiFetch` já faz hoje para toda chamada autenticada.
- **Validado com chamadas reais a `facebook.com` e `graph.facebook.com`, mesma limitação já documentada pro `MetaAdapter.sendMessage`** (2026-07-22): o botão real no navegador navegou de fato pro dialog OAuth da Meta (recebeu "ID do app inválido" - App de teste, esperado); o `callback` com `state` errado rejeitou por CSRF sem chamar a API; com `state` certo, chamou `oauth/exchange`, que bateu de verdade em `graph.facebook.com/oauth/access_token` e recebeu o erro real da Meta ("Missing or invalid client id"), propagado como mensagem clara até a tela. Confirma que a integração está correta ponta a ponta; falta só um App Meta aprovado de verdade pra fechar com sucesso.
- **Exclusão de workspace é soft-delete puro, num ponto único de bloqueio** (2026-07-22): `Workspace.deletedAt` (migration) + `WorkspacesService.softDelete` (só Owner, audit log `workspace.deleted`) + `listForUser` filtrando `deletedAt: null`. O bloqueio de acesso vive só no `WorkspaceRolesGuard` (`membership.workspace.deletedAt` checado ali, não em cada controller) - toda rota com `@WorkspaceRoles` já para de funcionar sozinha assim que o workspace é deletado, incluindo tentar deletar de novo (o guard rejeita antes do `softDelete` nem rodar). Anonimização de PII fica de fora - é soft-delete reversível, não o "direito ao esquecimento" completo.
- **Paginação por cursor em `GET /contacts` substitui offset/limit** (2026-07-22): `ContactsService.list` busca `take + 1` linhas com `cursor: {id}, skip: 1` (quando há cursor) e usa a linha extra só pra saber se existe próxima página, sem uma query de `count` separada. Cursor por `id` mesmo ordenando por `createdAt` funciona corretamente porque `id` é único - ao contrário de offset, não derrapa se linhas forem inseridas entre uma página e outra. Resposta `{items, nextCursor}`; sem UI pra atualizar porque não existe tela de contatos no frontend ainda.
- **Atalho `Cmd/Ctrl+S` no Flow Builder**: `useEffect` sem array de dependências (roda em todo render) só pra manter o listener de teclado sempre fechado sobre o `saveDraft` mais recente, que por sua vez fecha sobre `nodes`/`edges` atuais - simplificação deliberada em vez de memoizar `saveDraft` com `useCallback` e gerenciar suas dependências, dado o escopo pequeno da feature.
- **Fases D, E e F validadas contra a API/UI reais** (2026-07-22): `DELETE /workspaces/:id` real marcou `deletedAt`, sumiu de `GET /workspaces`, e uma tentativa seguinte (acessar ou deletar de novo) voltou 403 "This workspace has been deleted". Criei 5 contatos reais e paginei em 3 chamadas com `take=2`, usando o `cursor` devolvido a cada vez - sem repetir nem pular linha, última página com `nextCursor: null`. No navegador real, `Ctrl+S` no Flow Builder mostrou "Rascunho salvo." sem clicar em nenhum botão.
- **Fase A (infraestrutura gerenciada real) concluída - primeiro deploy de produção da API** (2026-07-22): `https://many-zac-api.onrender.com` está no ar contra Postgres (Supabase, projeto `unplzzkifgdjunfktvfj`, região `ca-central-1`) e Redis (Upstash, TLS) reais - não mais só localhost. O serviço `many-zac-api` no Render já existia de uma sessão anterior mas com as 10 env vars secretas vazias (por isso os deploys vinham falhando, inclusive um commit só de documentação); configuradas via API do Render e disparado o deploy. `curl /health` → `{"status":"ok","checks":{"database":"ok","redis":"ok"}}`; um signup + criação de workspace reais via HTTPS pública confirmaram o caminho de ponta a ponta, com o workspace de teste removido depois via o soft-delete da Fase D.
- **Gotcha real descoberto no deploy: conexão direta do Supabase é IPv6-only, Render não tem saída IPv6** - a `DATABASE_URL` original (`db.<projeto>.supabase.co:5432`) falhava com `P1001: Can't reach database server` só a partir do Render; funcionava via `psql` local porque a máquina de desenvolvimento tem conectividade IPv6 (um falso negativo que atrasou o diagnóstico). Corrigido trocando para o **Session pooler** do Supabase (`aws-0-<região>.pooler.supabase.com:5432`, usuário `postgres.<projeto>` em vez de `postgres` simples) - resolve por IPv4, mesmo comportamento de conexão direta (ao contrário do Transaction pooler na porta 6543, que não lida bem com prepared statements do Prisma sem configuração extra). README atualizado para recomendar Session pooler, não a conexão direta.
- **Roteamento de webhook por `externalAccountId` puro, sem pinar provider** (2026-07-22): achado de code review pré-App-Meta - `ChannelsService.processInboundWebhook` buscava a conexão por `(MetaAdapter.provider, externalAccountId)` com o provider fixo em `INSTAGRAM`, o que tornava qualquer conexão `MESSENGER` inalcançável por webhook (cairia em "unknown external account"), e o OAuth automático conectava uma Page do Facebook rotulada como `INSTAGRAM`. Corrigido: o lookup agora é só por `externalAccountId` (o espaço de IDs da Meta é global - Page id para eventos do Messenger, IG business account id para eventos do Instagram, nunca colidem), o provider gravado em `InboundEvent`/`ContactIdentity` vem da conexão encontrada (não mais do adapter), e o OAuth conecta a Page como `MESSENGER` (é o que uma Page id + Page token são). Validado contra a API real local: conexão `MESSENGER` criada via API, webhook assinado com `entry.id` da Page → `accepted: 1` (antes do fix, o mesmo cenário real retornou `unknownAccount: 1`), replay dedupado, e o contato criado com identidade `MESSENGER`. De quebra: `GET /contacts` ganhou desempate por `id` no `orderBy` (ordem determinística com `createdAt` empatado), `?take=` inválido virou 400 em vez de 500 do Prisma, e a troca de token OAuth valida a presença do `access_token` na resposta. 195/195 testes.
- **Conexão de canal agora inscreve a Page no webhook do App (`/{page-id}/subscribed_apps`)** (2026-07-22): descoberto ao configurar o App Meta real - a assinatura de webhook em nível de App (callback URL + fields, criada via `POST /{app-id}/subscriptions` com o app access token `id|secret`) só diz à Meta *onde* entregar; cada Page precisa **também** se inscrever no App, senão as DMs dela nunca chegam ao webhook. `MetaAdapter.subscribePageToApp` faz o `POST /{page-id}/subscribed_apps` com `subscribed_fields: ["messages"]` e o token da Page; `ChannelsService.connect` chama isso **best-effort** (try/catch com warning, campo `webhookSubscribed` na resposta) - não-fatal de propósito, para conexões manuais com token de teste (dev local, suíte de validação) continuarem funcionando; o warning no log é a pista do operador se mensagens reais não estiverem chegando. A assinatura de App, por sua vez, foi criada via Graph API direto (sem painel), e o handshake de verificação (`hub.challenge`) passou contra a API de produção real na primeira tentativa.
- **Configuração de deploy feita via API, não pelo painel** - a pedido do usuário, em vez de eu manusear senha de conta ou colocar segredo em arquivo versionado (`render.yaml` já marca as env vars secretas como `sync: false` de propósito), o usuário gerou uma API key do Render (escopo de conta) e um token da Management API do Supabase; usei ambos via `curl` direto (`api.render.com/v1`, `api.supabase.com/v1`) para listar/editar env vars, disparar deploy, ler logs e inspecionar config do projeto - nenhum segredo tocou o git, nenhuma senha de conta foi usada. Chaves salvas fora do repo em `~/.many-zac/render-api-key` e `~/.many-zac/supabase-management-token` (`chmod 600`), documentado em `docs/NEXT_STEPS.md` para sessões futuras.
